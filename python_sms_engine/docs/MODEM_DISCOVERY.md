Python SMS Engine — Modem Discovery Refactor Report
Problem Statement
A Python FastAPI service drives 5 Quectel EC25 USB modems on Linux. The original modem detection used /dev/serial/by-id symlink names to group interfaces and identify which port to use for SMS. This approach was unreliable — when multiple identical Quectel modems are connected, the by-id hash can map to the wrong physical device, causing SMS to be sent through the wrong modem or fail entirely.

Goal: Make modem discovery deterministic, efficient, and production-grade for SaaS use with 1000+ modems.

Root Cause Analysis
Why /dev/serial/by-id failed
The by-id names are generated by udev using a hash of USB descriptors. With 5 identical Quectel EC25 modems, those hashes collide or get suffixed inconsistently. The result was that interface grouping (pairing if02 with if03 for the same physical modem) was unreliable — the wrong ttyUSB numbers were being grouped together.

The correct source of truth
Linux sysfs USB topology. Every ttyUSB device has a symlink at /sys/class/tty/ttyUSBX that resolves to a real path containing the physical USB bus+port address and interface number:


/sys/devices/pci0000:00/0000:00:14.0/usb3/3-7/3-7.4/3-7.4.4:1.2/ttyUSB2/tty/ttyUSB2
                                                                      ↑   ↑
                                                              physical    interface=2 (if02)
The physical component (3-7.4.4) is unique per physical modem and stable across reboots.

What We Tried (Trial & Error)
Attempt 1 — by-id grouping
Approach: Parse /dev/serial/by-id/ filenames, extract interface tag (if02/if03), group by device hash.

Result: Failed. Identical modems produced colliding or wrong hashes. Wrong physical ports grouped together.

Attempt 2 — First sysfs implementation
Approach: Read /sys/class/tty/ttyUSBX symlinks, build groups by physical USB parent.

Bug hit: device_sysfs path was constructed as m.group(1) + m.group(2) which produced a doubled path like .../3-7.4.4/3-7.4.4. Reading idVendor from that path returned None, so the Quectel vendor filter rejected every device → 0 modems discovered.

Fix:


# WRONG — duplicates the physical directory name
device_sysfs = m.group(1) + m.group(2)

# CORRECT — m.group(1) already ends with "/" after the physical dir
device_sysfs = m.group(1).rstrip("/")
Attempt 3 — Probing both if02 and if03
Approach: After sysfs grouping, probe both if02 and if03 per modem at startup, pick the better-scoring one.

Result: Functionally correct but wasteful — 5 modems = 10 serial probes at startup. Also failed to store the fallback port properly when fallback logic was later needed in sms_service.py.

Attempt 4 — Stale __pycache__
Problem: After rewriting modem_detector.py and sms_service.py, uvicorn kept loading old compiled bytecode. Evidence: logs showed old [SYSFS GROUPS] / [USB MODEM] / [TRY] if03 format; new fields (modem_id, error_layer) were absent from responses.

Fix:


find . -name "*.pyc" -delete && find . -name "__pycache__" -type d -exec rm -rf {} +
Attempt 5 — File sync issue (Mac vs Server)
Problem: Claude Code runs on Mac at /Users/reg/Documents/Web Development/Projects/..., server runs at ~/Documents/WebDev/python-sms-engine/.... Edits on Mac weren't automatically on the server. Server kept running the intermediate version.

Fix: Copy updated files to the server, clear cache, restart.

Final Architecture
modem_detector.py — Sysfs-based, if02-only probing

/dev/ttyUSB*  →  sysfs symlink  →  physical USB parent + interface number
                                         ↓
                              filter: vendor == 2c7c (Quectel)
                              collect: if==2 → primary port
                                       if==3 → fallback port (stored, not probed)
                                         ↓
                              probe only if02 (1 probe per modem)
                              accept if: at_ok + sim_ready + creg_registered
                                         ↓
                              store: sim_id=IMSI, modem_id=IMEI, fallback_port=if03
Key improvement: 5 modems = exactly 5 probes. Previously was 10 (probing if02 + if03 per modem).

modem_registry.py — Warm refresh

TTL expired?
    └─ cache empty or force?  → full sysfs scan + probe
    └─ all ports still exist? → warm refresh (just update TTL, no I/O)
    └─ port disappeared?      → full rescan
Port existence check (os.path.exists()) is essentially free — no serial I/O. Full scans only happen at startup or when hardware changes.

at_client.py — Structured error codes

# Extracts numeric codes from modem responses:
# +CMS ERROR: 50  → cms_code=50 (no credit)
# +CME ERROR: 10  → cme_code=10 (SIM not inserted)

class SMSExecutionError(Exception):
    code: str           # e.g. "SEND_FAILED"
    raw: Optional[str]  # full modem response text
    cms_code: Optional[int]  # network/carrier layer
    cme_code: Optional[int]  # modem/equipment layer
sms_service.py — Error layer classification

SMSExecutionError
    └─ code in HARDWARE_ERRORS  → error_layer = "hardware"  (port dead, timeout)
    └─ cme_code is not None     → error_layer = "modem"     (SIM missing, PIN, etc.)
    └─ cms_code is not None     → error_layer = "network"   (no credit, carrier reject)
    └─ else                     → error_layer = "unknown"
Three-tier retry on send:

Primary port (if02)
Retry same port (after 0.5s sleep)
Failover to fallback_port (if03) from registry

What Remains Untested in Production
error_layer: "network" with real non-null cms_error_code — requires a SIM with no credit or a carrier reject
error_layer: "modem" with real non-null cme_error_code — requires a SIM hardware failure or missing SIM
Fallback port activation — requires forcing an if02 failure to confirm if03 is used and registry is updated
These will surface naturally in production.
